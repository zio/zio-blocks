The **ZIO-Golem SDK** is a type-safe Scala library for building agents on the Golem platform. It provides trait-based agent definitions, compile-time macro derivation, and seamless integration with Golem's durability, RPC, and persistence features. 

Core types: `BaseAgent`, `AgentDefinition`, `HostApi`, `GolemSchema`, `Transactions`, `Result`, and configuration helpers.

```scala
Define an agent interface
@agentDefinition
trait Counter extends BaseAgent {
  class Id(val name: String)
  def increment(): Future[Int]
  def get(): Future[Int]
}

// Implement the agent
@agentImplementation()
class CounterImpl(name: String) extends Counter {
  private var count = 0
  override def increment(): Future[Int] = Future.successful { count += 1; count }
  override def get(): Future[Int] = Future.successful(count)
}
```

## Introduction

This module brings **idiomatic Scala** to the Golem platform with zero boilerplate. Instead of writing RPC handlers, schema codecs, and WIT bindings by hand, ZIO-Golem uses Scala 3 macros to derive all of these at compile time from your agent trait.

If you're building a distributed, stateful application that needs durability guarantees, agent-to-agent communication, or transaction support, ZIO-Golem provides first-class abstractions for all of these patterns—all type-safe, all compile-checked.

## Motivation

Building agents on Golem traditionally requires:
- Defining trait interfaces (the agent contract)
- Writing serialization schemas for all types
- Implementing RPC request/response handlers
- Encoding WIT types and values
- Managing durability, snapshots, and transaction logic

ZIO-Golem **eliminates boilerplate** by deriving all of this from your trait definition using macros. You write the agent logic; the compiler generates the rest.

**Why Scala?**
- Type safety: catch bugs at compile time, not runtime
- Functional composition: agents as pure, composable functions
- Ecosystem: leverage the entire ZIO ecosystem (schema derivation, testing, concurrency)
- Multiplatform: Scala.js allows agents to run in the browser or on the server

## Installation

Add the ZIO-Golem SDK to your `build.sbt`:

```scala
libraryDependencies += "dev.zio" %%% "zio-golem-core"  % "@VERSION@"
libraryDependencies += "dev.zio" %%% "zio-golem-model" % "@VERSION@"
libraryDependencies += "dev.zio" %% "zio-golem-macros" % "@VERSION@"
```

Supported Scala versions: **2.13.x** and **3.8.x**. For Scala.js: use `%%%` instead of `%%`.

## Overview

The module has two main layers:

**`model`** — Pure types and schemas:
- `BaseAgent` — Base trait for agent interfaces
- `AgentDefinition` — Type-level metadata (name, durability mode, constructor)
- `GolemSchema` — Type class for encoding/decoding Scala types
- `StructuredSchema`, `StructuredValue` — Schema and value representations
- Annotations — `@agentDefinition`, `@description`, `@endpoint`, etc.
- Configuration — `Config`, `ConfigBuilder`, `Secret` for agent parameters
- Helpers — `Transactions`, `Result`, `Snapshotting`, `Principal`, `Uuid`

**`core`** — Runtime components:
- `AgentImplementation` — Register agent implementations (Scala.js)
- `AgentClient` — Connect to remote agents and invoke methods
- `HostApi` — Access Golem's host API (oplog, durability, persistence)
- `OplogApi`, `DurabilityApi`, `ContextApi` — Granular host API layers
- `Rdbms` — Relational database integration (MySQL, PostgreSQL)

## How They Work Together

Agent execution in ZIO-Golem follows this flow:

**1. Definition → Registration**
```
@agentDefinition
trait MyAgent extends BaseAgent
     ↓
AgentImplementation.registerClass[MyAgent, MyAgentImpl]
     ↓
Macro derives: RPC handlers, schema encoders, WIT types, metadata
```

**2. Invocation (await, trigger, schedule)**
```
val agent = MyAgent.get("instance-id")
     ↓
agent.someMethod()       // Await (blocks for result)
agent.trigger.someMethod() // Fire-and-forget
agent.schedule.someMethod(time) // Scheduled execution
```

**3. Persistence & Durability**
```
Method executes → HostApi tracks operations (oplog)
     ↓
On failure: Oplog rolls back, compensation runs
     ↓
On success: Snapshot saved (if configured)
```

**4. Data Flow**
```
Input (Scala type)
    ↓
GolemSchema.encode() → StructuredValue
    ↓
WIT serialization → Binary
    ↓
[Network or Host API]
    ↓
Binary → WIT deserialization
    ↓
GolemSchema.decode() → Output (Scala type)
```

**Core Relationships:**
- `BaseAgent` — All agents extend this; provides `agentId`, `agentType`, `agentName`
- `AgentDefinition[T]` — Describes an agent type T; generated by macros
- `AgentImplementation` — Registers implementations; generates RPC handlers
- `GolemSchema[A]` — Any type with a `Schema[A]` instance automatically gains a `GolemSchema[A]` via the implicit `GolemSchema.fromBlocksSchema[A]` — no explicit derivation step is needed
- `HostApi` — Accessed from within agent methods to control durability, persistence, retries
- `Transactions` — For atomic, multi-step operations with automatic rollback
- `Result[Ok, Err]` — WIT-compatible error type (like Rust's `Result`)

## Common Patterns

### Pattern 1: Simple Agent with Constructor

Define constructor parameters via an inner `class Id`:

```scala
import golem.runtime.annotations.{agentDefinition, description}
import golem.BaseAgent
import scala.concurrent.Future

@agentDefinition
trait Shard extends BaseAgent {
  class Id(val tableName: String, val shardId: Int)
  
  @description("Get a value from the table")
  def get(key: String): Future[Option[String]]
  
  @description("Set a value in the table")
  def set(key: String, value: String): Future[Unit]
}
```

When mounting over HTTP, constructor parameters must match path variables:
```scala
@agentDefinition(mount = "/api/{tableName}/{shardId}")
trait Shard extends BaseAgent {
  class Id(val tableName: String, val shardId: Int)
  // ...
}
```

### Pattern 2: RPC with Agent Companion

For ergonomic `.get()` and `.getPhantom()` access, define a companion object:

```scala
import golem.AgentCompanionBase

object Shard extends AgentCompanionBase[Shard]

// Now clients can do:
// val shard = Shard.get("table", 1)
// val phantom = Shard.getPhantom("table", 1, uuid)
```

### Pattern 3: Stateful Agent with Snapshots

Persist state across invocations using snapshots by mixing in the `Snapshotted` trait:

```scala
import golem.runtime.annotations.agentImplementation
import golem.Snapshotted
import zio.blocks.schema.Schema
import scala.concurrent.Future

case class CounterState(value: Int)

@agentImplementation()
class CounterImpl() extends Counter with Snapshotted[CounterState] {
  var state: CounterState = CounterState(0)
  val stateSchema: Schema[CounterState] = Schema.derived
  
  override def increment(): Future[Int] = Future.successful {
    state = state.copy(value = state.value + 1)
    state.value
  }
}
```

### Pattern 4: Transactions with Automatic Rollback

Execute multi-step operations atomically:

```scala
import golem.Transactions

val result = Transactions.infallibleTransaction { tx =>
  val step1 = Transactions.operation(
    (_: Unit) => Right(42)
  )((_, _) => Right(()))
  
  tx.execute(step1, ())
}
```

### Pattern 5: Configuration & Secrets

Agents can declare configuration fields:

```scala
import golem.config.ConfigSchema
import golem.runtime.annotations.agentDefinition
import golem.BaseAgent

@agentDefinition
trait ConfiguredAgent extends BaseAgent {
  val apiKey: String
  val timeout: Int = 30
}
```

Configuration is provided by the Golem runtime, overridable per deployment.
