Async — cross-runtime benchmarks

Throughput and per-operation allocation for ZB Async vs scala.concurrent.Future, Kyo, and Cats Effect — on the eager / ready path each runtime drives synchronously, and on genuinely-suspended work completed off-thread (the True async section).

How to read this. Throughput is higher-is-better (k/s = thousand, M/s = million, B/s = billion ops/sec). Bars are linear and normalized to the fastest runtime in each row — ZB Async is the fastest on every operation, so it is a full 100% bar throughout and the competitors render as thin slivers because they are genuinely 10×–10,000× slower on the eager path (read the numeric labels for the exact magnitude). In the True async section the bars are much closer — there the cross-thread wakeup dominates, not the runtime overhead, so the field narrows to within ~3×. The allocation badge is gc.alloc.rate.norm (bytes allocated per op, lower-is-better); ≈0 B means escape analysis scalar-replaced everything.
  • Fairness. Every runtime is driven to a value synchronously on the eager path: ZB .block, Cats Effect .unsafeRunSync(), Kyo .eval. Future runs on ExecutionContext.parasitic (callbacks inline, no thread hop) with the value read via f.value.get.get — i.e. Future's best case; a real thread-pool executor would be substantially slower.
  • Residual ZB allocations are the unavoidable user-facing objects: Either/Failure (16 B), the Tuple3 from zip3 (56 B), and the Completer behind promise (16 B).
  • mapError is bounded by the user's new RuntimeException(cause) stack-trace fill-in across all four runtimes — not library overhead.
  • Collapsed rows. Operator families with the same relative profile — and the n=10/100/1000 size variants — are shown as one representative row; the grey suffix lists the members it stands in for (e.g. zipWith · zipLeft · zipRight). Genuine outliers are kept separate: zip3 (allocates the user Tuple3) and mapError. Kyo has no apples-to-apples promise/sync-callback analog (—).
