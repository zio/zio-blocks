# route-pattern-typed-vars learnings

- A jj-created workspace still needs git worktree metadata for sbt/git discovery in this repo. The minimal fix was to create the workspace root `.git` pointer plus the matching admin files under the main repo's `.git/worktrees/<workspace-name>/` directory.
- `SegmentCodec.PathVars` had to remain a separate phantom track from the existing runtime value type `A`; leaf segments can carry the named phantom marker directly, while the non-leaf composition cases stay neutral for this todo.
- The Scala 2/3 test split works best as separate files under `scala-2/` and `scala-3/`, with the shared backward-compat regression test left in the unsuffixed test source tree.

## Todo 2 findings (Combined/Transform PathVars propagation)

- `SegmentCodec.Combined[A,B,C]`'s `left`/`right` fields are typed as the plain, unrefined `SegmentCodec[A]`/`SegmentCodec[B]` (not the richer `WithBoundaries[..]{type PathVars=..}` refinement), so `left.PathVars`/`right.PathVars` are ALWAYS abstract/unreduced at the `Combined` case class's own definition site, regardless of what expression is written there - exactly like `Prefix`/`Suffix` are simple projections, but a real CONCATENATION needs actual computation, which a plain projection can't do. Consequence: `Combined`'s own class-body `type PathVars = ...` declaration can never be "correct" in an externally-observable sense; the REAL, precise, positionally-correspondent concatenation only exists on the `~` extension method's own declared/refined RETURN TYPE (`WithBoundaries[C,P,S2]{type PathVars=PVC}`), produced by capturing the operands' PathVars via a NEW type parameter on the extension itself (`self: WithBoundaries[A,P,S]{type PathVars=PV}`) - the exact same "dependent-type-capture via refinement on the extension/implicit-class receiver" pattern already used for `P`/`S` (BoundaryTag). This means `Combined`'s class-body PathVars member is written as a documented best-effort/placeholder (Scala 3: a real match type `L match { case EmptyTuple => R; case h *: t => h *: Concat[t,R] }`, unbounded so it applies to the abstract `left.PathVars`; Scala 2.13: literally just `(L, R)`, since there is no way to compute anything more precise there) - only the `~` method's refined return type is what the acceptance tests (and any future Todo 3 consumer) actually observe.
- The NEW endpoint-scoped combinator (`PathVarTuples`, one file per platform, never touching `zio.blocks.combinators.Tuples.Tuples`) needs TWO pieces per platform: (1) a type-level `Concat[L,R]` used only for the `Combined` class-body placeholder discussed above, and (2) a term-level typeclass `Combine[L,R]{type Out}`/`Combine.WithOut[L,R,O]` (Scala 3: one `given instance[L,R]: WithOut[L,R,Concat[L,R]]`; Scala 2.13: a whitebox macro mirroring `Tuples.TuplesMacros.tuplesImpl`'s `isTuple`/`tupleElements`/arity-extension approach, but concatenating BOTH sides' elements instead of appending a single new one, with `Unit` (`NoPathVars`) as the identity element on either side) - added as a NEW implicit/`using` parameter on the `~` extension method (both overloads: segment~segment and segment~literal), NOT as a new constructor parameter on `Combined` (constructor arity is frozen by the plan).
- Adding the new implicit `pathVarsCombiner: PathVarTuples.Combine.WithOut[PV, PV2, PVC]` parameter to `~` triggered `-Ywarn-unused` + `-Xfatal-warnings` ("parameter ... is never used") on Scala 2.13 MAIN sources (not suppressed by the existing `-Wconf:cat=unused&src=.*/test/.*` rule, which only covers test sources) even though the pre-existing `canCombine: CanCombine[S, P2]` implicit param is ALSO never referenced in the method body and does NOT trigger the same warning - never fully root-caused why `canCombine` is exempt, but the pragmatic, zero-behavior-change fix is to explicitly reference the new implicit via `val _ = pathVarsCombiner` as the first line of the method body (a compile-time-only evidence parameter, safe to discard).
- `SegmentCodec.literal(value)` rejects `/` (and any character requiring URL encoding) - do NOT use `literal("/")` as a separator in `~`-chain test fixtures (it throws `IllegalArgumentException` at macro-expansion time on Scala 3, which then cascades into a confusing secondary `AssertionError: Cannot get tree of no symbol` from `boundaryInfo`); use a plain literal like `literal("-")` instead.
- `CanCombine`'s numeric-adjacency rule (`Cannot combine two numeric segments`) means a same-name-collision test like `int("id") ~ int("id")` must have a non-numeric literal segment interposed (`int("id") ~ literal("-") ~ int("id")`) to type-check at all.
- On Scala 2.13, `SegmentCodec.int("id").transform(identity, identity)` fails to type-infer (`found: Nothing => Nothing, required: Int => Int`) - needs an explicit type argument, `transform[Int](identity, identity)`; Scala 3's improved local type inference handles the bare `identity, identity` form fine in the same test.
- `sbt --client` interactive PTY sessions in this environment intermittently DROPPED or TRUNCATED input mid-string when a full command + newline was written in one `pty_write` call (observed twice: once truncating `"++3.8.3; endpointJVM/test"` to just `"++3.8.3;"` with the rest silently lost, another time to `"++3.8"`), leaving the sbt shell's main thread genuinely idle (confirmed via `jstack`) waiting for a command that never arrived - NOT a slow compile. Ctrl+C in that state kills the whole client+server process group (server does NOT survive detached). The reliable workaround: skip the interactive PTY shell entirely and invoke `sbt --client -Dsbt.color=false "++<version>; <task>"` as a single ONE-SHOT non-interactive command via the plain command-runner tool (it connects to any already-running server and is not subject to the interactive-input-drop issue) - this is also simpler and matches the AGENTS.md command template more directly than a hand-fed interactive shell.
