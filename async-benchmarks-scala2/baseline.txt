Scala 2 macro direct-style `.await` benchmark gate — BASELINE
=============================================================

Module : async-benchmarks-scala2  (Scala 2.13.18, JVM only)
Suite  : zio.blocks.async.bench.AsyncScala2MacroBench
Purpose: prove the GENERATED CODE SHAPE / allocation profile of the Scala 2
         `Async.async { ... .await ... }` macro (internal.AsyncMacros). This is
         NOT a cross-runtime throughput comparison (Kyo / Cats Effect are
         deliberately absent); it compares the macro's output to hand-written
         `flatMap` controls in the same module, with `-prof gc`.

Host   : Apple M3 Ultra, macOS (Darwin 25.3.0, arm64)
JDK    : OpenJDK 26.0.1 (Homebrew)

Command (allocation gate):
  sbt "++2.13.18; async-benchmarks-scala2/Jmh/run -p n=10 -wi 3 -i 3 -f 1 -prof gc \
       zio.blocks.async.bench.AsyncScala2MacroBench"

Note on settings: `gc.alloc.rate.norm` (bytes/op) is DETERMINISTIC (±0.001 B/op
across iterations), so a short run pins it exactly; throughput (ops/s) carries
normal JMH variance and is reported only as an order-of-magnitude sanity check.
`n` only varies the `while`/`flatMap`-chain rows; the HOF rows use fixed-size
collections, so a single `n` is representative for them.

INTERPRETATION
--------------

1. Scalar `.await` is ZERO-ALLOCATION and matches the hand-written control.
   `zb_asyncAwait1` (≈1e-5 B/op, ~1.06e9 ops/s) is identical to `zb_flatMap1`
   and `zb_map1`; `zb_asyncAwaitSeqVals` is likewise ≈0. The Scala 2 macro emits
   a `flatMap` chain that escape analysis folds away exactly like a hand-written
   one — no hidden continuation or `Pollable` allocation for ready values.

2. A `var` mutated ACROSS `.await` inside a `while` allocates a FIXED, constant-
   in-`n` ref-cell cost (`zb_asyncAwaitWhileVarN` ≈1e-4 B/op), the same order as
   the hand-written `zb_flatMapControlN`. This is the inherent CPS cost of
   threading mutable loop state through a suspension point; it is per-block, not
   per-link. (Matches the Scala 3 / DCA finding in async-benchmarks/baseline.txt.)

3. HOF emitters allocate only their RESULT collection plus a builder, with no
   per-element continuation garbage for ready awaits:
     - option: 16 B/op (a single `Some`).
     - foldLeft: 104 B/op (no result collection — accumulator threading only).
     - list/vector/map/collect: 184-312 B/op (cons cells / builder + result).
     - Array.filter: 136 B/op (`Array.newBuilder` + result array).
     - Array.flatMap: 224 B/op (builder drain + `ArraySeq.unsafeWrapArray`
       normalization of the raw-array closure result).
     - Array.map: 360 B/op — the EAGER strategy materializes an intermediate
       `List[Async[B]]` and a `List[B]` (via `collectAll`) before `toArray`. This
       is the highest HOF row; it is correct and bounded, and is a candidate for
       a future allocation optimization (build straight into an `Array` builder
       instead of routing through `collectAll`'s `List`). Tracked as a possible
       follow-up; not a correctness or per-link-cost issue.

GATE VERDICT: PASS. The primary criterion — scalar `.await` collapses to the
hand-written `flatMap` control with zero per-link allocation — is met decisively.
HOF rows allocate only their designed result; the Array paths (incl. primitive
element preservation and raw-array flatMap normalization) behave as designed.
